# AWS deployment — push to ECR + 3 ASGs (manual)

This walks through a manual, single-operator flow that does not
require any IaC tool: push the local image to ECR, then stand up three
one-instance Auto Scaling Groups, one per AZ. Each instance pulls the
image from ECR on boot and runs `merutable-node` under systemd.

## Artifacts in this directory

```
deploy/aws/
├── README.md                     # this doc
├── ecr-push.sh                   # tag + push linux/amd64 image to ECR
├── render-user-data.sh           # produce per-AZ user-data scripts
├── user-data/
│   ├── user-data.sh.tpl          # AL2023 user-data template
│   └── node-{1,2,3}.user-data.sh # generated by render-user-data.sh
└── iam/
    ├── README.md
    ├── trust-policy.json         # EC2 assume-role trust
    └── ecr-read-policy.json      # least-privilege ECR pull
```

## Prerequisites

- AWS CLI v2 authenticated against the target account / region
- Docker image already built locally:
  `./deploy/scripts/build.sh` (produces `merutable-node:local`)
- A VPC with **three subnets**, one per AZ, that can reach ECR
  (public subnets work out of the box; private subnets need an ECR
  VPC endpoint or a NAT gateway)
- A Security Group that allows inbound **TCP 9100** from the other
  two subnets' CIDRs (so the pods can speak Raft/gRPC to each other),
  plus any ingress rule you need from your own machine

## Step 1 — push the image to ECR

```bash
export AWS_REGION=us-west-2
./deploy/aws/ecr-push.sh
```

Sample output ends with something like:

```
==> done
    image URI: 123456789012.dkr.ecr.us-west-2.amazonaws.com/merutable-node:latest
```

Capture that URI — you'll need it in Step 3.

```bash
export ECR_IMAGE=123456789012.dkr.ecr.us-west-2.amazonaws.com/merutable-node:latest
```

## Step 3 — reserve static private IPs (one per AZ)

We're doing manual peer discovery: each node has to know the other
two nodes' addresses at boot time. EC2's default DHCP address isn't
stable, so we pin three private IPs inside the VPC.

Two common options:

1. **Allocate three ENIs** (one per subnet) with explicit
   `--private-ip-address`, attach each via the Launch Template. This
   is what the rendered user-data assumes.
2. **Reserve by subnet-range convention** — e.g. the first usable IP
   in each subnet — and set `--private-ip-addresses` on the Launch
   Template's network interface.

Example for option 1 (one per AZ):

```bash
SG_ID=sg-0123...          # your merutable SG
SUBNET_AZ1=subnet-aaaa    # in AZ-a
SUBNET_AZ2=subnet-bbbb    # in AZ-b
SUBNET_AZ3=subnet-cccc    # in AZ-c

aws ec2 create-network-interface \
  --subnet-id $SUBNET_AZ1 --groups $SG_ID \
  --private-ip-address 10.0.1.10 --description merutable-node-1

aws ec2 create-network-interface \
  --subnet-id $SUBNET_AZ2 --groups $SG_ID \
  --private-ip-address 10.0.2.10 --description merutable-node-2

aws ec2 create-network-interface \
  --subnet-id $SUBNET_AZ3 --groups $SG_ID \
  --private-ip-address 10.0.3.10 --description merutable-node-3
```

```bash
export NODE1_IP=10.0.1.10
export NODE2_IP=10.0.2.10
export NODE3_IP=10.0.3.10
```

## Step 5 — create three Launch Templates

One per node. Key settings:

| Field                | Value                                          |
|----------------------|------------------------------------------------|
| AMI                  | Amazon Linux 2023 (x86_64, kernel 6)           |
| Instance type        | `t3.medium` or larger                          |
| IAM instance profile | `merutable-node-ec2`                           |
| Security group       | your merutable SG (allows inbound 9100)        |
| Network interface    | the pre-created ENI (step 3) for that AZ       |
| User data            | paste `node-<N>.user-data.sh` contents         |
| Storage              | root EBS (30+ GiB); `/data` lives on it        |

CLI variant (node-1 shown):

```bash
LT_NAME=merutable-node-1
AMI_ID=$(aws ssm get-parameters \
  --names /aws/service/ami-amazon-linux-latest/al2023-ami-kernel-default-x86_64 \
  --query 'Parameters[0].Value' --output text)

aws ec2 create-launch-template \
  --launch-template-name $LT_NAME \
  --launch-template-data "$(cat <<JSON
{
  "ImageId": "$AMI_ID",
  "InstanceType": "t3.medium",
  "IamInstanceProfile": { "Name": "merutable-node-ec2" },
  "UserData": "$(base64 -i deploy/aws/user-data/node-1.user-data.sh)",
  "NetworkInterfaces": [
    {
      "DeviceIndex": 0,
      "NetworkInterfaceId": "eni-xxxxxxxxxxxxxxxxx",
      "DeleteOnTermination": false
    }
  ],
  "TagSpecifications": [
    { "ResourceType": "instance",
      "Tags": [
        { "Key": "Name",            "Value": "merutable-node-1" },
        { "Key": "app",             "Value": "merutable" },
        { "Key": "merutable-az",    "Value": "AZ-1" },
        { "Key": "merutable-node",  "Value": "1" }
      ] }
  ]
}
JSON
)"
```

Repeat for `merutable-node-2` and `merutable-node-3` with their
respective ENIs and user-data files.

## Step 6 — create three ASGs (min=max=desired=1, one per AZ)

One per Launch Template. The `min=max=desired=1` pattern gives you
self-healing (an unhealthy instance gets replaced) without accidental
scale-out that would re-hash the ring.

```bash
for N in 1 2 3; do
  case $N in
    1) SUBNET=$SUBNET_AZ1 ;;
    2) SUBNET=$SUBNET_AZ2 ;;
    3) SUBNET=$SUBNET_AZ3 ;;
  esac
  aws autoscaling create-auto-scaling-group \
    --auto-scaling-group-name merutable-asg-$N \
    --launch-template "LaunchTemplateName=merutable-node-$N,Version=\$Latest" \
    --min-size 1 --max-size 1 --desired-capacity 1 \
    --vpc-zone-identifier "$SUBNET" \
    --health-check-type EC2 \
    --tags "Key=Name,Value=merutable-node-$N,PropagateAtLaunch=true"
done
```

The ASG boots an instance which attaches the pinned ENI, the
user-data script runs, `merutable-node` comes up on port 9100, and
systemd keeps it healthy.

## Teardown

```bash
for N in 1 2 3; do
  aws autoscaling delete-auto-scaling-group \
    --auto-scaling-group-name merutable-asg-$N --force-delete
  aws ec2 delete-launch-template \
    --launch-template-name merutable-node-$N
done

aws ec2 delete-network-interface --network-interface-id eni-...   # x3

# ECR repo + IAM profile — see iam/README.md
aws ecr delete-repository --repository-name merutable-node --force
```

## Known limitations (important)

- **`merutable-node` binary hard-codes the self-address to
  `0.0.0.0:<port>`** (`crates/merutable-cluster/src/bin/merutable-node.rs`).
  That address gets written into the metadata Raft group as this
  node's address and will be handed back to peers unchanged — so
  cross-AZ `connect_peer` calls will fail. Related to **GAP-10** in
  `docs/PHASE1_RAFT_CLUSTER.md`. The infra here stands each node up
  independently (you can gRPC into each one on `10.0.N.10:9100` and
  exercise the single-node path), but multi-node Raft replication
  will not work end-to-end until the binary accepts an `--advertise
  <ip:port>` flag (or equivalent) and uses it in `ClusterConfig.this_node.address`.
- **Multi-node Raft wiring gaps remain** (GAP-1, GAP-11, GAP-19 in
  the same design doc). Even if the advertise-address issue is
  fixed, writes that need to be proxied across AZs won't replicate
  through a real Raft group yet.
- **No real AZ failure-domain isolation** beyond the label — your
  ring AZ labels (`AZ-1`, `AZ-2`, `AZ-3`) and EC2 physical AZs
  (`us-west-2a`, etc.) are correlated by subnet choice but not
  enforced by the code.
- **`emptyDir`-equivalent state** — the container writes to `/data`
  which is on the root EBS volume, gone when the instance is
  replaced. Mount a dedicated EBS volume in the user-data (or swap
  to EFS) when persistence matters.
